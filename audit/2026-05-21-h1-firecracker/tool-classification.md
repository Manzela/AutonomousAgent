# H1 Firecracker — Tool Classification

**Purpose:** Decide which existing and planned tools land on the Firecracker tier vs stay on `shell_sandbox` / `cloud_sandbox`. This is the input to the new `match:` rule in `config/toolsets.yaml` and the gate that determines whether a Firecracker microVM is spun up for a given call.

**Source of truth for current tools:** `config/toolsets.yaml` (existing routes) + survey of `hermes-agent/tools/`, `lib/`, and tools registered via MCP servers (`config/mcp/`).

---

## 1. Classification axes

Two axes, applied independently:

**Axis A — Code trust:**
- **Trusted** — code was written by Anthropic / project owner. Sandboxing is defense-in-depth, not adversarial defense.
- **Untrusted** — code was generated by an LLM, supplied by an operator, or received from a non-Anthropic A2A peer. Sandboxing is the security boundary.

**Axis B — Side-effect footprint:**
- **Read-only** — cannot mutate host state, cannot exfiltrate.
- **Local-write** — mutates host filesystem (writable scratch only).
- **Network-bound** — makes outbound HTTPS to allowlisted hosts.
- **Arbitrary** — anything goes (write FS + network + CPU/RAM-bound work).

The cell (Axis A × Axis B) → tier:

|              | Read-only | Local-write | Network-bound | Arbitrary |
|--------------|-----------|-------------|---------------|-----------|
| Trusted      | `in_process` | `shell_sandbox` | `external_https` / `browser_sandbox` | `shell_sandbox` |
| **Untrusted**| `shell_sandbox` (still!) | **`firecracker_sandbox`** | **`firecracker_sandbox`** | **`firecracker_sandbox`** |

The rule is **untrusted + any side-effect = Firecracker**. Untrusted read-only is still `shell_sandbox` because the cost of spinning a microVM for a `grep` is not worth the marginal isolation improvement over Docker. (This is the "right tool for the right job" principle from ADR-0003.)

## 2. Tool-by-tool inventory

| Tool match | Current tier | Trust | Side-effect | Tier (post-H1) | Change? |
|---|---|---|---|---|---|

| `shell`, `bash`, `git`, `jq`, `run_command` | `shell_sandbox` | Trusted | Local-write + network (constrained) | `shell_sandbox` | No |

### 2.2 Tools that do not exist yet but are on the H1+A2A horizon

| Planned tool | Source | Trust | Side-effect | Tier |
|---|---|---|---|---|
| `a2a_peer_exec` (call into a peer agent's tool) | A2A spike (#38) | **Untrusted** (peer is non-Anthropic) | Arbitrary | `firecracker_sandbox` |

| `wasm_eval` (CPU-bound untrusted regex/eval) | Not on roadmap | Untrusted | CPU-bound | (deferred — WASM is its own tier discussion) |

**Decision rule for new tool additions.** Any tool whose handler takes a string and treats it as code-to-be-executed lands on `firecracker_sandbox` by default. The exception requires an ADR (this raises the cost of accidental escalation).

### 2.3 Tools that look untrusted but aren't

These are tools that consume LLM-generated content but do NOT execute it as code, so they stay on lighter tiers. Documented here to prevent over-sandboxing:

| Tool | Why not Firecracker |
|---|---|
| `write_file` | Writes a string to a host path. The string isn't executed. Side-effect is captured by `shell_sandbox` scratch-dir constraints. |
| `edit_file` | Same as above. |
| `markdown_render` | Pure read-write of strings. No code-execution surface. |
| `judge_*` (4-judge consensus) | Calls LLMs over HTTPS. Trusted (Anthropic endpoint), network-bound → `external_https`. |
| `kanban_*` | Local DB mutation, trusted. `in_process` or `shell_sandbox`. |

## 3. The default tier still matters

`config/toolsets.yaml:18` sets `default_tier: shell_sandbox`. After H1, **this stays the same** — the default is NOT `firecracker_sandbox`. Reasoning:

- Untrusted code-execution tools have explicit names (`run_python`, etc.) that will match their explicit route.
- If an unknown tool name lands at the router, the safer move is `shell_sandbox` (Docker + cap-drop) than `firecracker_sandbox` (extra ~800ms boot, $$$). Auto-escalating unknown tools to the heaviest tier would create a cost foot-gun.
- The "default-deny" posture is preserved by the existing `shell_sandbox` default — it already provides hard FS and network restrictions.

## 4. New `config/toolsets.yaml` route (the actual config change)

Adds one route after the existing `cloud_sandbox` route, *and* changes the existing route's tier. Net diff:

```diff
   # arbitrary code execution
-  - match: ["run_python", "run_javascript", "exec_code", "code_interpreter"]
-    tier: cloud_sandbox
+  - match: ["run_python", "run_javascript", "exec_code", "code_interpreter", "a2a_peer_exec", "a2a_peer_python"]
+    tier: firecracker_sandbox
     evaluate_after: true
```

`cloud_sandbox` stays defined in the tier preamble (for back-compat), but is no longer referenced by any active route. ADR-0010 documents the move; a future cleanup can remove the unused tier definition.

## 5. New `firecracker_sandbox` tier definition

Add to the `Tiers` preamble (`config/toolsets.yaml:1-12`):

```yaml
#   firecracker_sandbox — KVM microVM via Firecracker, kernel-isolated,
#                         no host network, ephemeral rootfs, jailer-isolated.
#                         Use for untrusted code execution (LLM-generated,
#                         A2A-peer-supplied, operator-supplied).
```

## 6. Evaluator-panel posture

`evaluate_after: true` stays for all `firecracker_sandbox` routes. The 4-judge consensus cost is dwarfed by the microVM boot cost (~$0.0001 per judge call vs ~$0.001 per Firecracker invocation), and the consensus signal is most valuable on exactly these untrusted-code execution calls.


The last test is the load-bearing one: it locks the policy that unknown tools do not auto-escalate.
